# 安全ルール・Claude Code 固有ルール

## 編集前の確認手順

1. 対象ファイルを必ず読む（編集前に現在の内容を確認）
2. 変更の影響範囲を把握する（何が変わるかを言語化してから実行）
3. 可逆性を確認する（元に戻せない操作は事前にユーザーへ確認）

---

## 操作の許可レベル

### Always（確認不要）

- ファイルの読み取り・検索（Read / Grep / Glob）
- 単一ファイルの型チェック・lint・フォーマット
- git status / git log / git diff などの参照系操作
- テスト実行（単一ファイル・単一テスト）

### Ask first（実行前に確認）

- `git push`（リモートへの反映）
- `git reset --hard`（変更の破棄）
- ファイルの削除
- GitHub PR / Issue の作成・コメント
- 外部サービスへのリクエスト（Slack、メール等）
- CI/CD パイプラインの変更
- システムパッケージのインストール（`apt install` 等）
- フルビルド・E2E テストの実行
- `source ~/.zshrc`（意図しない副作用の可能性）
- `kill -9` / `pkill` の広範囲実行
- `chmod -R` / `chown -R` の未確認実行

### Never（絶対に実行しない）

- `~/.ssh/` 以下の編集・削除
- `~/.gnupg/` 以下の編集・削除
- `~/.secrets/` 以下のファイルを `git add`
- シークレット・API キーをファイルや応答に平文で書く
- `rm -rf` を引数の確認なしに実行
- `eval` に未検証の動的文字列を渡す
- `/mnt/` 以下の Windows ファイルを削除
- `.env` ファイルを git でトラッキング

---

## 絶対に編集しないファイル

| ファイル | 理由 |
|---|---|
| `~/.zshrc` | 変更するとシェルが起動しなくなる可能性がある |
| `~/.bashrc` / `~/.bash_profile` / `~/.profile` | 同上 |
| `~/.config/mise/config.toml` | 誤変更で環境が壊れる |
| `~/.config/sheldon/plugins.toml` | 変更後 `sheldon lock` が必要 |
| `~/.gitconfig` | Git のグローバル設定 |
| `~/.ssh/` 以下すべて | 認証情報 |
| `~/.gnupg/` 以下すべて | GPG 鍵 |

---

## シークレット管理

- シークレットは必ず `~/.secrets/` 以下に保存する
- `~/.secrets/env/` — 環境変数形式（例: `export API_KEY=xxx`）
- `~/.secrets/gcp/` — GCP サービスアカウント JSON
- `~/.secrets/` 以下は git 管理外・パーミッション 700/600 を維持する
- 値を dotfiles や設定ファイルに直書きしない
- 環境変数はパスのみ設定ファイルに記載し、値は `~/.secrets/` から読み込む
- 新しいシークレットが必要な場合は保存先を必ずユーザーに確認する

## Claude Code 固有ルール

### ツール優先順位

専用ツールを Bash より優先する。

| 操作 | 使うツール | 使わない |
|---|---|---|
| ファイル読み取り | `Read` | `cat`, `head`, `tail` |
| ファイル編集 | `Edit` | `sed`, `awk` |
| ファイル作成 | `Write` | `echo >`, `cat <<EOF` |
| ファイル検索 | `Glob` | `find`, `ls` |
| 内容検索 | `Grep` | `grep`, `rg` |

### ファイル操作

- 既存ファイルを編集する前に必ず `Read` で内容を確認する
- 不要なファイル（README、ドキュメント類）を勝手に作成しない

### コード生成

- 過剰な実装をしない（求められた以上の機能追加・コメント追加を避ける）
- エラーハンドリングは実際に起こりうるケースのみ実装する
- セキュリティ脆弱性（SQLi・XSS・コマンドインジェクション等）を含むコードを生成しない

### git 操作

- コミットはユーザーから明示的に求められた場合のみ行う
- コミットメッセージは変更の「why」を中心に書く
- `--no-verify` / `--no-gpg-sign` は使わない
- `git add` は個別ファイル指定を優先（`-A` `.` は避ける）

### 応答スタイル

- 応答は短く簡潔に
- ファイルパスは `パス:行番号` 形式で示す
- 絵文字は使わない（明示的に求められた場合のみ）

### メモリ管理

- セッション横断で記憶すべき情報は `~/.claude/projects/` の memory ファイルに保存する
- 「覚えておいて」と言われた場合のみ永続化する

### 新規プロジェクト作成時

`/new-project` スキルを使う。

---

## 不明瞭な指示への対応

- 指示の意図が不明な場合は実行前に確認する
- 「とりあえずやってみる」で破壊的操作を行わない
- エラーが出た場合は原因を調べてから対処する（同じコマンドをリトライしない）
